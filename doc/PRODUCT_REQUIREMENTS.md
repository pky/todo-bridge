# TodoBridge プロダクト要件定義

## 1. 背景

Remember The Milk の置き換えとして、自分の運用に合ったタスク管理環境を Web 中心で再構築したい。あわせて、家族と Todo を共有できる構造を持たせ、関心テーマごとに必要な情報を集めて、あとで読む・対応する Todo へ接続できるようにしたい。

## 2. 解決したい課題

- RTM の既存タスクを移行しやすくしたい
- 日常的に使うタスク管理を、自分で制御できる構成にしたい
- 家族と Todo を共有できるようにしたい
- Web から素早く操作できることを優先したい
- 将来の iOS 展開も見据え、データモデルとバックエンドを先に整えたい
- AI / iOS / Android など、自分や家族に必要なテーマの情報を効率よく収集したい
- 初回設定などで関心テーマを選び、必要な情報だけを表示したい
- テーマごとに適切な更新頻度を変え、情報量とノイズのバランスを取りたい
- 集めた情報を「あとで読む」「対応する」タスクへ繋げたい
- 家族で共有したい記事や関心テーマを Todo と同じ文脈で扱いたい
- 変化が速い IT 系テーマは高頻度で、変化が比較的遅いテーマは週次中心で扱いたい

## 3. 目標

- Web 版で日常利用に耐えるタスク管理機能を提供する
- RTM エクスポートからの移行を可能にする
- 家族共有へ拡張しやすいデータ構造を維持する
- Google ログインで安全に利用開始できる
- 個人利用を前提に、応答速度とデータ整合性を両立する
- 初回設定で関心テーマを選び、そのテーマに応じた情報を表示できる
- AI / iOS / Android に加えて、将来的に育児や金融のようなテーマも扱える
- テーマごとに更新頻度を変えられる
- IT 系テーマは高頻度更新、それ以外は週次中心という運用方針を持てる
- 情報をタスク管理と連携しやすい形で扱える
- Firebase を共通バックエンドとして、将来の iOS 版展開に繋げる

## 4. 非目標

- 初期段階で RTM の全機能を完全再現すること
- パスワード認証など複数認証方式の提供
- 初期段階で大規模マルチテナント SaaS のような運用を目指すこと
- すべての計画書や内部調査メモを public 向け資料にすること

## 5. 想定ユーザー

- 主利用者: 開発者本人
- 将来の拡張利用者: 家族共有や複数端末利用を想定する小規模ユーザー
- 利用環境:
  - Web ブラウザ
  - 将来的な iOS アプリ

## 6. 想定利用シーン

- 既存の RTM データを取り込んで、そのまま日常のタスク管理を継続する
- 自分用のタスクを Web から素早く追加、確認、完了する
- 将来的な家族共有を見据えて、Todo を共有できる構造で管理する
- 初回設定で関心テーマを選び、自分や家族に必要な情報だけを確認する
- AI / iOS / Android 関連情報をまとめて確認し、重要な更新を見逃しにくくする
- 将来的には育児や金融のようなテーマも選べるようにする
- テーマに応じて、毎日更新が必要な情報と週次で十分な情報を分けて受け取る
- IT 系テーマは日次に近い頻度で確認し、それ以外は週次で要点を把握する
- 気になった情報を「あとで読む」「対応する」タスクへ繋げる
- 家族で共有したい記事や話題を Todo と同じ流れで扱う

## 7. 機能要件

### 7.1 コア要件

- タスクの作成、更新、完了、削除ができる
- リスト単位でタスクを管理できる
- タグ、期限、開始日、ノートなどの主要属性を持てる
- RTM エクスポート JSON を読み込める
- Google ログインとログアウトができる

### 7.2 拡張要件

- スマートリストで条件別にタスクを抽出できる
- 初回設定や設定画面で関心テーマを選択できる
- 情報機能で AI / mobile のようなテーマ別トピックを扱える
- 将来的に育児、金融などのテーマを追加できる
- テーマごとに更新頻度や収集頻度を調整できる
- AI / iOS / Android などの IT 系テーマは高頻度更新を前提にし、それ以外は週次中心で扱える
- 情報からタスクへ繋げやすい導線を持てる
- 設定画面からニュースや通知周りの挙動を調整できる
- 将来的に家族共有や iOS 展開へ拡張できる構造を維持する

## 8. 非機能要件

### 8.1 パフォーマンス

- 初期表示は過度に待たせない
- タスク追加や状態変更は体感上すぐ反映されることを重視する
- データ量が増えても主要操作が破綻しない

### 8.2 セキュリティ

- Firebase Authentication と Security Rules でアクセス制御する
- 原則としてユーザーは自分のデータのみ扱える
- 秘密情報はリポジトリに含めない
- 認証方式は、明確な理由と互換性検証なしに変更しない
- ログイン導線は `Mac Chrome`、`iPhone相当ブラウザ`、`iOS PWA` の再ログイン互換を維持する

### 8.3 コスト

- テーマ数が増えても、AI 利用量と Firebase 利用量が過度に増えないことを優先する
- 無料枠で継続運用しやすい構造を維持する
- すべてのテーマを同じ頻度で収集せず、更新頻度や優先度で負荷を制御する

### 8.4 保守性

- フロントエンドと Functions の責務を分離する
- 影響範囲を絞った変更と検証ができる構造を維持する
- AI を補助に使っても、最終判断と品質責任は人間が持つ
- 認証・router・PWA 関連変更時は、認証回帰テストを必ず通す

## 9. 制約

- バックエンドは Firebase を前提とする
- 認証は Google ログイン中心で進める
- Web 版先行で開発する
- AI と Firebase の利用量は、無料枠で収まりやすい運用を優先する
- public リポジトリとして見せるため、内部資料と公開資料を分けて管理する

## 10. 受け入れ条件

- Web 版で主要なタスク管理フローが完結する
- RTM データのインポートができる
- Google ログイン後に主要画面へ到達できる
- `pnpm --prefix web test:auth` が通る
- `Mac Chrome`、`iPhone相当ブラウザ`、`iOS PWA` の少なくとも1回の再ログイン確認が完了している
- `web` と `functions` のビルドが通る
- 公開資料だけでもプロダクトの意図と構成が理解できる

## 11. 未決事項

- iOS 版の実装範囲と着手タイミング
- 家族共有機能の公開範囲
- ニュース機能の今後の優先度
- 通知やカレンダー連携の公開方針
