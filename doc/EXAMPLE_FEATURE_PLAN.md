# 実装計画の例: タスク追加のオプティミスティック UI

## 位置づけ

このドキュメントは、TodoBridge で実際に行っている `設計 -> タスク分解 -> 実装 -> 検証` の流れを公開向けに要約したものです。内部の作業ログをそのまま出すのではなく、AI を使ってどのように整理し、実装へ落としているかが分かる代表例としてまとめています。

## 背景

タスク追加時、Firestore への書き込み完了を待ってから UI を更新していたため、回線が遅い環境では「ボタンを押しても反応が遅い」状態になっていました。

## 目標

- タスク追加直後に UI へ反映する
- Firestore 書き込み失敗時は安全にロールバックする
- 既存のタスク追加フローを大きく壊さない

## 設計方針

採用した方針は `Optimistic UI + Retry` です。

1. Firestore のドキュメント ID を事前生成する
2. ローカル state に即時追加して UI を先に更新する
3. バックグラウンドで Firestore へ保存する
4. 一時的な通信失敗はリトライする
5. 最終的に失敗した場合だけロールバックして通知する

この方針により、UX を改善しつつ、失敗時の整合性も維持できます。

## タスク分解

### Task 1: ストアの追加処理を optimistic update 対応にする

- 対象: `web/src/stores/tasks.ts`
- 目的: UI を先に更新し、保存は非同期で後追いする
- 完了条件: タスク追加直後に一覧へ反映される

### Task 2: リトライ処理を分離する

- 対象: `web/src/utils/retry.ts`
- 目的: Firestore 保存失敗時の再試行を共通化する
- 完了条件: 一時的なネットワーク失敗に対して再試行できる

### Task 3: 失敗時のユーザー通知を追加する

- 対象: `web/src/components/ToastNotification.vue`, `web/src/App.vue`
- 目的: 最終失敗時だけ明確に通知する
- 完了条件: ロールバック時にトーストが表示される

## 検証方針

- まずタスクストアのユニットテストで状態遷移を確認する
- 次に関連ユーティリティのテストを確認する
- 最後に `web` の型チェックとビルドで破壊がないことを確認する

想定コマンド:

```bash
cd web
npm run test:run
npm run type-check
npm run build
```

## AI の使い方

この種の変更では、AI に以下を担当させます。

- 問題の整理
- データフローの比較
- 失敗時の分岐パターンの洗い出し
- 最小差分での修正案作成
- 必要な検証項目の列挙

一方で、以下は人間が判断します。

- どの UX を優先するか
- ロールバックの条件
- どのエラーをリトライ対象にするか
- 差分が既存設計と矛盾しないか

## 実装後に得られること

- 変更前より操作レスポンスが体感上速くなる
- 通信失敗時の挙動が明確になる
- 設計意図と実装タスクの対応関係が追いやすくなる
