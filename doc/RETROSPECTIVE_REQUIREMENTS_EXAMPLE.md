# Retrospective 要件定義の例: タスク追加の高速化

## 1. 背景

タスク追加時に Firestore 書き込み完了まで UI 反映を待っていたため、ネットワーク状態によっては操作が重く感じられていた。

## 2. 解決したい課題

- タスク追加ボタンを押してもすぐに一覧へ出ない
- 追加に失敗した時の挙動が分かりにくい
- 遅延が UX にそのまま出てしまう

## 3. 目標

- タスク追加直後に UI に反映される
- 一時的な通信失敗では無駄に操作失敗と見せない
- 保存が最終的に失敗した場合だけ明確にロールバックする

## 4. 非目標

- 完全なオフラインモードの実装
- すべての操作を同時に optimistic update 化すること
- Firestore 依存をなくすこと

## 5. 想定ユーザー

- 日常的にタスクを追加する本人
- モバイル回線や不安定なネットワーク下で操作する利用者

## 6. ユーザーストーリー

- ユーザーとして、タスクを追加した瞬間に一覧へ表示されてほしい
- ユーザーとして、通信が一瞬不安定でも追加操作をやり直したくない
- ユーザーとして、本当に失敗した時だけ分かりやすく通知されてほしい

## 7. 機能要件

- タスク追加時にローカル state を先に更新する
- Firestore 保存はバックグラウンドで行う
- 一時的な失敗時は再試行する
- 最終失敗時はロールバックして通知する

## 8. 非機能要件

- 既存のタスク ID 管理と整合が取れていること
- 既存画面や他のタスク操作を大きく壊さないこと
- 失敗時の状態不整合を残さないこと

## 9. 制約

- Firestore を継続利用する
- 既存の `tasks` store の責務を大きく変えない
- UI 全体の大規模改修はしない

## 10. 受け入れ条件

- タスク追加後、UI に即時反映される
- Firestore 保存成功時に二重登録や表示ズレが起きない
- 失敗時にロールバックされる
- 関連テスト、型チェック、ビルドで回帰がない

## 11. 設計への橋渡し

この要件から、実装では以下の論点に分解できる。

- ID をどのタイミングで確定するか
- 楽観更新と永続化をどう分離するか
- どのエラーを再試行対象にするか
- 失敗時にどの UI で通知するか
