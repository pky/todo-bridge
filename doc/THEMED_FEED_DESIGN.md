# テーマ選択型情報機能 設計メモ

## 1. 設計の前提

要件は [THEMED_FEED_REQUIREMENTS.md](./THEMED_FEED_REQUIREMENTS.md) を前提とする。

現在の実装は `ai` や `mobile` のような固定トピック前提で成り立っているが、将来的にテーマ数が増えることを考えると、固定タブだけでは拡張しづらい。

## 2. 設計方針

- `topic` を固定値ではなく、定義データとして扱う
- 初回起動時にテーマ選択を行う
- UI は「少数テーマならタブ」「全体は一覧やセレクタ」で扱う
- テーマごとに更新頻度を持たせる
- 情報項目から Todo への導線を明示する
- AI と Firebase の利用量が無料枠で収まりやすい構造を優先する

## 3. 情報設計

### 3.1 テーマ定義

テーマは定義データとして持つ。

想定属性:

- `id`
- `label`
- `category`
- `updateFrequency`
- `isHighFrequency`
- `shareScope`
- `defaultSelected`

例:

- `ai`
- `ios`
- `android`
- `parenting`
- `finance`

### 3.2 ユーザー設定

ユーザーごとに以下を保持する。

- 選択済みテーマ
- テーマごとの並び順
- テーマごとの更新頻度上書き
- 家族共有するテーマかどうか

## 4. 画面設計方針

### 4.1 初回起動

- 関心テーマ選択画面を用意する
- 候補から複数選択できる
- 高頻度テーマと低頻度テーマが分かる説明を付ける

### 4.2 ホーム

- 選択済みテーマの情報を統合表示する
- 高頻度テーマを優先表示する
- Todo へ変換しやすい操作を近くに置く

### 4.3 テーマ別表示

- 少数テーマは上部タブで切り替える
- テーマ数が多い場合は「すべてのテーマ」一覧またはセレクタを使う
- したがって、タブは主 UI の一部であって、唯一のナビゲーションにはしない

## 5. データモデル方針

### 5.1 テーマ定義

- `appConfig/topics/{topicId}` または同等の定義を持つ
- ここに表示名や更新頻度を置く

### 5.2 ユーザー設定

- `users/{uid}/topicPreferences/{topicId}`
- 選択状態、並び順、頻度上書きなどを保存する

### 5.3 情報項目

- 既存の `topics/{topic}/articles` 構造を活かしつつ、テーマ追加可能な形に一般化する
- 記事は `topicId` を必須で持つ

## 6. 更新戦略

- 高頻度テーマ:
  - 日次またはそれに近い頻度
- 低頻度テーマ:
  - 週次中心
- 収集ジョブは頻度グループごとに分ける

これにより、すべてのテーマを毎日更新せずに済む。

## 7. コスト制御方針

- 全テーマを毎回収集しない
- 高頻度テーマと低頻度テーマで収集ジョブを分ける
- 低頻度テーマは週次中心でまとめて処理する
- AI による要約や分類は、必要なテーマや優先記事に絞る
- Firebase の読み取りは、選択済みテーマと必要期間に限定する
- 初回表示で全テーマ分のフィードを同時に読み込まない

## 8. Todo 連携

情報項目ごとに以下の操作を想定する。

- あとで読む
- 対応する
- 家族に共有する

Todo へ落とす際は、元記事 URL や要約を持ったタスクを生成できるとよい。

## 9. 今の実装から見直すべき箇所

- 固定トピック前提の型
- 固定ルート前提の画面構成
- タブだけでトピックを切り替える前提
- 収集頻度が固定されている前提
- 情報項目と Todo の接続が弱い箇所

## 10. 実装優先順

1. テーマ定義モデルの追加
2. 初回テーマ選択と設定画面
3. テーマ定義に基づく UI 切り替え
4. 更新頻度の分離
5. Todo 連携の強化
